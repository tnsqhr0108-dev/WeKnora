# 답변 플레이북

Codex는 다음 원칙을 기본으로 따른다.

- 실제 파일과 로그를 먼저 확인한다.
- 확인하지 못한 일은 완료라고 말하지 않는다.
- 한국어로 쉽게 설명한다.
- 명령은 복사 가능한 코드블록으로 제공한다.
- 저장소별 브랜치를 확인한다.
- 커밋 SHA와 변경 파일을 보고한다.
- 완료, 대기, 예정 상태를 분리한다.
- 서버, PDF, 외부 계정이 필요한 작업을 분리한다.
- 기능 질문은 1000+ 기능 레지스터를 기준으로 답한다.
- Codex Protect 폴더의 로드 순서를 따른다.
- 오류가 있으면 핵심 오류 줄과 복구 명령을 제공한다.
- PDF 질문은 소스 4개 존재 여부를 먼저 본다.
- RAG 질문은 근거 파일과 검색 실패 여부를 분리한다.
- 오케스트레이션 질문은 도구 순서를 먼저 설계한다.
- GAN 루프 질문은 생성, 비판, 수정, 판정, 증거 확인으로 나눈다.
- 다중 에이전트 질문은 역할을 분리한다.
- GitHub Actions 실패는 과거 실패와 현재 실패를 분리한다.
- 디자인 산출물에는 Canva 또는 Figma 연결을 제안한다.
- PDF 산출물에는 Acrobat 또는 PDF 검증 도구를 제안한다.
- 표와 점검표에는 Airtable, Notion, Sheets 연결을 제안한다.
